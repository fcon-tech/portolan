# Product Specification: Semantic Component Investigation Contract

> **Status:** draft for admiral review.
>
> **Authority:** `08-portolan-product-charter.md` governs product identity,
> ontology, trust, and navigation. `15-atlas-reading-experience.md` and
> `16-atlas-drilldown-decision-semantics.md` remain valid only where they do
> not reduce Portolan back to a repository/component graph.

## Reader And Goal

Reader: an implementation agent or reviewer taking the next Portolan demo
slice after `16`.

Post-read action: implement a reusable component-investigation contract where
Portolan can open a selected component and explain its place in the system, why
it exists, how it is structured, what makes it risky or unusual, and how it
overlaps or contrasts with adjacent parts of the landscape.

This is not a slice about Solr, Spark, Flink, or any other named technology.
Those names are examples only. The product goal is the portable investigation
capability.

## Product Failure Being Fixed

The current demo can pass all structural harnesses while still failing the
product:

- component pages mostly repeat repository metadata, family labels, and
  external links;
- C4 shows an honest-empty Container level and a promoted-unit list, but not a
  meaningful decomposition of the system;
- clicking a component does not explain its ecosystem role, purpose, internal
  model, operational quirks, or relationship to adjacent capabilities;
- components can coexist as graph nodes without the atlas explaining semantic
  overlaps, duplicated concepts, or alternative ways to solve the same system
  problem;
- tests prove that surfaces exist, but not that the admiral learns anything
  beyond what is visible from a repository list;
- broken reader artifacts such as `[object Object]`, `undefined`, `Object not
  found`, and generic "not found" detail pages are not caught as demo failures.

The failure is not primarily visual. It is an ontology and acceptance failure:
the implementation treated repository/component nodes as the main product
object. They are not. The main product object is the **semantic system
landscape**.

## Decision Gate

**Simpler/Faster:** keep the current clean-stack shell and generated local HTML.
Add a reusable semantic investigation layer and prove it on a small selected
sample of components instead of replacing the whole atlas, introducing a
database, adding a hosted service, or attempting full-corpus semantic extraction
in one slice.

**Blocking Edge Cases:** Bigtop is a distribution, not a single application;
many components are upstream OSS projects with their own internal architectures;
local corpus evidence may show package recipes and manifests but not runtime
behavior; some useful ecosystem knowledge may come from documentation or
curated knowledge rather than the checked-out corpus; agent-produced semantic
assertions must therefore be separated from deterministic facts and labeled by
source boundary.

**Existing Open Source:** existing tools solve parts of the problem:

- Backstage's catalog model gives teams typed entities such as systems,
  components, APIs, resources, domains, and ownership. Portolan should learn
  from the clarity of typed catalog objects, but should not become a manually
  maintained service catalog.
- Structurizr/C4 treats diagrams as views over a model. Portolan should learn
  from multi-view architecture mapping, but should not fabricate C4 containers
  from repository names.
- Sourcegraph/SCIP, GitHub code navigation, and OpenGrok are strong source
  navigation tools. Portolan should link to source anchors when available, but
  its value is not "jump to definition"; it is a readable expedition atlas.
- CodeQL path explanations show why an analysis result exists. Portolan should
  make component relationships and overlaps explainable paths, not anonymous
  graph edges.

The Portolan wedge remains: local-first, read-only, generated expedition bundle
that combines system meaning, evidence boundaries, and next navigation steps
for an admiral and agents.

## Core Product Correction

Portolan must answer semantic questions before repository questions.

A component investigation fails if it can be generated by listing repositories,
labels, and external links. It must explain:

1. where the component sits in the ecosystem;
2. why it exists;
3. how it is structured internally at a useful architectural level;
4. how it integrates with neighboring capabilities;
5. what overlaps, duplicates, or competes with it;
6. what is risky, unusual, legacy-shaped, or operationally important;
7. which claims are grounded in local corpus evidence, which are curated
   knowledge, which are agent hypotheses, and which are not assessed.

Repository evidence, manifests, package recipes, docs, and source anchors are
supporting material. They are not the atlas.

## Sample Policy

Implementation must prove this contract on a **small selected sample** of
components from Bigtop or another local corpus. The sample is not the product.

The sample must include:

- at least one component with a clear ecosystem capability;
- at least one pair of components with a meaningful semantic overlap or
  alternative capability;
- at least one component whose local corpus evidence is incomplete, forcing an
  honest source-boundary display.

Named examples such as Solr, Spark, and Flink may be used if they are useful,
but acceptance must not be written around those exact names or hardcoded facts.
The reviewer must be able to swap the sample for another eligible set without
rewriting the product contract.

## Owned Scope

Implement a narrow but reusable vertical slice:

- a semantic investigation data contract for selected components;
- a component investigation page or panel that renders the contract;
- an ecosystem placement map or panel that shows capability regions, not just
  repository nodes;
- semantic overlap/alternative relations between selected components;
- source-boundary handling for local corpus evidence, curated knowledge, agent
  hypotheses, and not-assessed gaps;
- browser-level acceptance that checks the reusable contract on the selected
  sample;
- negative acceptance checks that fail generic, broken, or content-poor pages.

## Explicit Non-Goals

- Do not build a full Bigtop encyclopedia in this slice.
- Do not hardcode the product around Solr, Spark, Flink, or any fixed set of
  OSS technologies.
- Do not require runtime execution, Docker, network probes, or package builds.
- Do not replace the whole viewer.
- Do not introduce a daemon, database, hosted service, or credentialed network
  integration.
- Do not claim runtime truth from local static evidence.
- Do not mark curated/public knowledge as deterministic corpus evidence.
- Do not hide honest gaps, but also do not let "not assessed" become the whole
  product experience.

## Semantic Ontology

This slice introduces a semantic layer over the existing atlas.

### Ecosystem

The whole target landscape as a system-of-systems. For Bigtop, this may be a
distribution ecosystem: upstream components, package recipes,
deployment/provisioning surfaces, tests, and runtime assumptions.

### Capability

What the ecosystem can do. Examples include search/indexing, processing,
querying, storage integration, coordination, security/governance,
workflow/orchestration, packaging, and deployment.

Capabilities are not repositories. A component can provide multiple
capabilities. Multiple components can overlap on one capability.

### Component

A named technology or subsystem that provides capabilities. A component may be
backed by a repository, package recipe, manifest entry, docs, source code,
runtime evidence, or curated knowledge. The investigation contract must not
assume that a component is equivalent to a repository.

### Concept

An internal architectural idea inside a component. Concepts help the admiral
understand how a component is built without diving into source code first.

For example, a search component may expose concepts such as index, schema,
query, shard, replica, and coordination. A processing component may expose
concepts such as job graph, scheduler, executor, state, checkpoint, connector,
or cluster manager integration. These are examples only, not acceptance terms.

### Semantic Relationship

A meaning-bearing relation between components, capabilities, or concepts.
Required relation types for this slice:

- `provides_capability`;
- `depends_on`;
- `integrates_with`;
- `overlaps_with`;
- `contrasts_with`;
- `packaged_by_target`;
- `not_assessed`.

`overlaps_with` is not a dependency edge. It means two components cover similar
problem space or duplicate architectural capability.

### Source Boundary

Each semantic assertion must be labeled:

- `local-corpus`: derived from the inspected target files/artifacts;
- `curated-knowledge`: stable curated knowledge about the component or domain;
- `agent-hypothesis`: agent inference, with or without cited evidence;
- `not_assessed`: the atlas cannot say yet.

`curated-knowledge` is allowed because the goal is system understanding, not
only file inventory. It must still carry a source card: official docs, project
docs, a checked-in curated note, or another resolvable artifact. Do not let
"the model knows this" become an invisible source.

The UI must make the boundary readable without forcing the admiral to inspect
raw evidence first.

## Component Investigation Contract

Opening an investigated component must show these sections. The exact visual
layout is flexible, but the information architecture is not.

### 1. Ecosystem Placement

Answer:

- What region of the ecosystem is this component in?
- What neighboring capabilities surround it?
- Which components are adjacent, upstream, downstream, overlapping, or unknown?

### 2. Purpose And Capabilities

Answer:

- Why is this component useful in the ecosystem?
- Which capabilities does it provide?
- What kind of workload or system need does it address?

### 3. Internal Model

Answer:

- What are the main concepts inside the component?
- How do those concepts relate?
- What should an admiral understand before asking an agent to modify or
  investigate this component?

This is not source-code documentation. It is a conceptual architecture map.

Each investigated component must have at least five internal concepts with a
plain-language explanation and source-boundary label, unless the page explicitly
states that the internal model is `not_assessed` and gives a concrete next
producer needed to assess it. A selected sample cannot pass if every component
uses the `not_assessed` escape hatch.

### 4. Integration Surface

Answer:

- How does the target ecosystem include, package, call, deploy, configure, or
  otherwise depend on this component?
- Which manifests, recipes, docs, tests, APIs, or deployment surfaces mention
  it?
- What dependencies or runtime expectations are visible?
- What is not assessed?

If exact source anchors are missing, the page must say so and name the next
producer needed.

### 5. Problems, Risks, And Peculiarities

Answer:

- What makes this component hard to operate, integrate, upgrade, or reason
  about?
- What version skew, packaging, runtime, configuration, coordination, migration,
  duplication, or conceptual complexity matters?
- What is known vs inferred?

Each investigated component must show at least two component-specific risks or
peculiarities. Generic text such as "inspect source" or "runtime not assessed"
does not count as a risk.

### 6. Overlap And Alternatives

Answer:

- Which components provide overlapping capabilities?
- Where do they duplicate concepts?
- Where are they meaningfully different?
- Why would the ecosystem contain both?

The selected sample must include at least one bidirectional overlap or
alternative relation. Opening either side of the pair must show the other side
and explain the overlap dimensions.

### 7. Evidence And Confidence Boundary

Answer:

- Which statements came from the local corpus?
- Which statements came from curated knowledge?
- Which are agent hypotheses?
- What cannot be verified in this run?

The page must not collapse these into one confidence badge. The admiral should
be able to read the main story first, then inspect evidence boundary when
needed.

### 8. Next Expedition

Answer:

- What should the next agent do to improve this component investigation?
- Which probe or producer would turn useful unknowns into evidence?
- What would change if that probe succeeds?

This section must contain actions tied to the component's semantic gaps, not
generic "inspect source" text.

## Ecosystem Placement Map

The atlas must include at least one semantic ecosystem map or panel that shows
capability regions and component placement.

The map may be rendered as cards, lanes, a matrix, or a lightweight diagram.
It must not be the existing repository graph with different colors.

Minimum behavior:

- show capability regions;
- place the selected sample components into one or more regions;
- show at least one visible overlap/alternative relation;
- link from a placed component to its investigation page;
- make source boundaries or not-assessed areas visible enough that the map does
  not imply false completeness.

## Data Contract

The implementation may use either an extension inside the generated navigation
bundle or a sidecar artifact. The generated data must be machine-readable and
must include this shape:

```yaml
semantic_investigation:
  sample:
    selection_reason: "Why these components prove the contract."
    components:
      - component:<id>
  components:
    - id: component:<id>
      display_name: Component Name
      ecosystem_regions:
        - capability:<region-id>
      purpose:
        summary: "<why this component exists in the ecosystem>"
        source_boundary: curated-knowledge
        source_ref: source:<id>
      capabilities:
        - id: capability:<id>
          label: Capability label
          source_boundary: local-corpus
          source_ref: evidence:<id>
      internal_concepts:
        - id: concept:<id>
          label: Concept label
          explanation: Plain-language explanation.
          source_boundary: curated-knowledge
          source_ref: source:<id>
      integration_surfaces:
        - kind: manifest | package_recipe | api | docs | tests | deployment | not_assessed
          label: Surface label
          source_boundary: local-corpus
          evidence_ref: evidence:<id>
      semantic_relations:
        - type: overlaps_with
          target_id: component:<other-id>
          dimensions:
            - capability:<id>
          explanation: Why this is an overlap, contrast, or dependency.
          source_boundary: agent-hypothesis
          source_ref: source:<id>
      risks:
        - id: risk:<id>
          explanation: Component-specific risk or peculiarity.
          source_boundary: agent-hypothesis
          source_ref: source:<id>
      evidence_boundary:
        local_corpus:
          - evidence:<id>
        curated_knowledge:
          - source:<id>
        agent_hypotheses:
          - risk:<id>
        not_assessed:
          - gap:<id>
```

YAML is acceptable for authoring fixtures, but generated bundles should remain
easy for the existing JS stack to parse and validate. If YAML is used, the plan
must justify parser/dependency choice or convert YAML to JSON at build time.

## UX Contract

The page must read like an atlas investigation, not a database record.

Required visible affordances:

- a concise top summary answering "what is this and why does it matter?";
- an ecosystem placement section above raw evidence;
- concept cards or a concept map for internal structure;
- overlap/alternative cards with links to adjacent component pages;
- risk/peculiarity cards;
- evidence boundary summary;
- next-expedition actions.

Forbidden visible outcomes:

- `[object Object]`;
- `undefined`;
- `Object not found` from any first-class visible link;
- generic "Named repository component present" as the main explanation;
- a component page whose only actionable content is external project/tracker/wiki
  links;
- a C4 or semantic map that is only a renamed repository graph.

## Acceptance Scenarios

### Scenario: Selected Component Teaches Ecosystem Meaning

Given the atlas is generated
And the semantic investigation sample contains at least one selected component
When the admiral opens a selected component
Then the page explains the component's ecosystem placement
And it explains why the component exists
And it shows at least five internal concepts or an explicit component-specific
not-assessed boundary with a concrete next producer
And it shows local integration evidence or a precise `not_assessed` boundary
And it shows at least two component-specific risks or peculiarities
And it contains no `[object Object]`, `undefined`, or generic repository-only
copy as the primary explanation.

### Scenario: Selected Pair Shows Bidirectional Semantic Overlap

Given the semantic investigation sample contains a selected overlap pair
When the admiral opens either component in the pair
Then the page shows the other component as an overlapping or alternative
component
And the overlap explains at least three dimensions
And the relation is bidirectional.

### Scenario: Component Page Separates Meaning From Evidence

Given a semantic component page contains curated knowledge and local corpus
evidence
When the admiral reads the page
Then the main story is readable without opening raw evidence
And each claim group exposes whether it is `local-corpus`,
`curated-knowledge`, `agent-hypothesis`, or `not_assessed`
And every curated-knowledge claim links to a resolvable source card.

### Scenario: Ecosystem Map Is Not A Repo Graph

Given the atlas is generated
When the admiral opens the semantic ecosystem map
Then the map shows capability regions
And selected components are placed in those regions
And at least one overlap/alternative relation is visible
And the map is not merely the existing repository graph with renamed labels.

### Scenario: Broken UI Artifacts Fail Acceptance

Given the generated atlas is opened in a browser
When every first-class visible link on semantic investigation pages is clicked
Then no destination shows `[object Object]`
And no destination shows `undefined`
And no destination shows `Object not found`
And no destination falls back to a generic component/repository page for any
selected component.

## Harness Requirements

Add a browser-level harness for this slice. Static string checks are not
sufficient.

The harness must:

- generate the review HTML for the selected target;
- read the generated semantic investigation sample from machine-readable data;
- open every selected component in a real browser;
- assert required section headings and minimum content counts;
- assert at least one bidirectional overlap/alternative pair;
- click all first-class links from the selected component pages;
- fail on `[object Object]`, `undefined`, `Object not found`, and generic
  repository-only main copy;
- record screenshots for the selected component pages and ecosystem placement
  map;
- write a machine-readable acceptance artifact mapping each manual review
  question to the selector/section/source-boundary evidence that answers it.

The harness may still print a human UX boundary, but it must not pass if the
page is obviously content-poor or broken.

## Manual Review Gate

This slice is not accepted until a cold reviewer can answer these questions
from the generated HTML without opening GitHub or the raw source tree:

1. Where does the selected component sit in the ecosystem?
2. Why is it useful?
3. How is it conceptually structured?
4. What are its visible risks or peculiarities?
5. Which selected components overlap or act as alternatives?
6. Where are the overlap pair similar and different?
7. Which claims are local corpus evidence versus curated knowledge,
   hypotheses, or not assessed?

If the reviewer says "I could have learned this by opening the repository list",
the slice fails.

## Implementation Notes

Start with fixture-backed semantic investigations if needed. That is acceptable
for a demo slice only if:

- the source boundary is visible;
- every curated-knowledge claim links to a resolvable source card;
- the data shape is the same shape future producers will generate;
- the harness proves the UI consumes generated data rather than hard-coded DOM
  text;
- the next-expedition section names the producer work needed to replace or
  enrich fixtures.

Do not let fixture-backed content become invisible product debt. The atlas must
say what is curated and what came from the inspected corpus.

## Rejected Alternatives

### Build Hardcoded Pages For Named Example Technologies

Rejected. Named technologies can illustrate the product, but if acceptance is
written around fixed facts about those examples, the implementation will become
a demo encyclopedia rather than a Portolan capability.

### Fix Only The Existing Component Dossier

Rejected. It would patch `[object Object]` and add a few fields, but it would
keep repository/component nodes as the main ontology. That repeats the same
failure.

### Generate Shallow Pages For Every Component

Rejected for this slice. It optimizes coverage over meaning and will likely
produce many generic pages. Build a high-quality investigation contract first,
then scale.

### Build Complete C4 Extraction First

Rejected for this slice. C4 remains useful, but it is not enough. The admiral
needs semantic ecosystem understanding, not only a decomposition diagram.

### Replace Portolan With Backstage/Structurizr/Sourcegraph

Rejected. These tools are important references, but they do not provide the
local-first expedition artifact Portolan is targeting. Portolan should pack or
link their kinds of outputs where useful, not disappear into any one of them.

## Verification Commands

Expected command set after implementation:

```bash
cd portolan-core && npm test && npm run test:deps
bash scripts/harness-atlas-navigation-index.sh
bash scripts/harness-atlas-reading-experience.sh
bash scripts/harness-atlas-drilldown-semantics.sh
bash scripts/harness-atlas-semantic-component-investigation.sh
```

Passing the first three harnesses is not sufficient. The new semantic
component-investigation harness is the acceptance gate for this slice.

## Done Means

- The selected sample is declared in machine-readable generated data with a
  selection reason.
- Every selected component has a semantic investigation page.
- At least one selected pair has bidirectional overlap/alternative relations.
- The ecosystem map shows capabilities, not only repositories.
- Component investigations include purpose, placement, internal concepts,
  integration surfaces, risks/peculiarities, overlap/alternatives, evidence
  boundary, and next expedition.
- Browser harness catches broken links and broken rendered values.
- Curated knowledge is visibly source-bound and resolvable.
- No selected component page contains `[object Object]`, `undefined`, or
  generic repository-only copy as the main product content.
- The result teaches a cold reader something material about the system that a
  repository list does not.
