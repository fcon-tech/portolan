# Spec 069 Review: Bigtop Architecture Synthesis

## findings:

### 1
- severity: **major**
- evidence: `stress/cursor-architecture-synthesis-output-2026-06-02.md` & `reviews/architecture-synthesis-ledger-2026-06-02.md` — C4 and C9 assessed `cannot_verify`; C1, C2, C5, C6, C8 assessed `partial`
- issue: The ledger records honest degraded states, which is good. However, FR-007 requires "three assessed independent non-GPT lanes or degraded replacements." The review lists Cursor as lane 1 and Portolan static analysis as lane 2, but the third lane is not clearly identified or is conflated with lane 1. If all three lanes share Cursor-produced output, the independence requirement is structurally unmet even at degraded-replacement level.
- recommendation: Name each lane explicitly (e.g., Lane A = ctags/symbol extraction, Lane B = Semgrep/pattern analysis, Lane C = Cursor composer synthesis), document which original evidence each lane consumed, and note that Lane C depends on Lanes A+B outputs. Mark FR-007 as "partial — two independent static lanes plus one synthesis lane consuming their output" rather than claiming three independent lanes.

### 2
- severity: **major**
- evidence: FR-003 — "run Cursor Agent composer-2.5 with Portolan evidence and record prompt/output"; `stress/cursor-architecture-synthesis-prompt-2026-06-02.md`
- issue: The stress prompt file is the Cursor Agent input, but there is no verifiable chain showing the prompt was actually fed to composer-2.5 versus authored manually and attributed. For a docs/stress slice that cannot run services, the minimum verification is a reproducibility assertion: commit hash, timestamp, model identifier, and a statement that output is unedited or edited with diff recorded.
- recommendation: Add a frontmatter block to both prompt and output files with `model: composer-2.5`, `timestamp`, `commit`, and `editing: none|diff-attached`. Without this, the evidence chain has a gap that a future auditor cannot close.

### 3
- severity: **minor**
- evidence: FR-002 — "include specs 059-068 evidence"; `docs/specs/069-bigtop-architecture-synthesis/spec.md`
- issue: The spec references prior evidence but does not enumerate which specific artifacts from 059-068 were consumed, making it impossible to confirm completeness without cross-reading all ten prior specs.
- recommendation: Add an annex table mapping spec 059-068 → artifact path(s) actually ingested by 069, with any gaps noted.

### 4
- severity: **minor**
- evidence: FR-005 — "do not start services, contact Kubernetes, install releases, mutate targets, or add network-dependent producers"; spec.md and plan.md
- issue: The spec correctly disallows these actions, but no verification gate exists confirming compliance. A post-hoc assertion in the ledger ("no services started") is present but not independently checkable.
- recommendation: Add a single checklist line in tasks.md confirming each prohibited action with a dated sign-off. This is trivial for a docs slice but closes the audit trail.

### 5
- severity: **minor**
- evidence: `reviews/architecture-synthesis-ledger-2026-06-02.md` — allowed wording section
- issue: The allowed/disallowed wording lists are well-drawn. However, the phrase "verified bounded producer outputs beyond Syft/CycloneDX for static deployment models" is close to an overclaim if "verified" is read as end-to-end verification rather than "verified bounded" (i.e., verified as bounded, not fully verified). Ambiguity risk.
- recommendation: Reword to "evidence-bounded producer outputs, verified as bounded (not fully verified), beyond Syft/CycloneDX for static deployment models" to eliminate the ambiguity.

### 6
- severity: **minor**
- evidence: FR-006 — "record whether human/enterprise parity is verified, partial, blocked, or unverified"
- issue: The ledger records this per-criterion but does not produce a single consolidated parity statement as FR-006 appears to require.
- recommendation: Add one top-level parity line: "Human/enterprise parity: unverified — insufficient runtime evidence, no paired human baseline run in this slice."

## not_assessed:

- **Runtime topology correctness** — cannot be assessed in a docs/stress slice per FR-005; deferred to a future runtime-privileged spec.
- **Post-wave paired full-rubric rerun (C8 gap)** — acknowledged in ledger as not performed; not verifiable here.
- **Actual composer-2.5 invocation fidelity** — no tooling available to confirm model identity; accepting stress output at face value per slice constraints.
- **Cross-spec 059-068 evidence completeness** — would require reading all ten prior spec artifacts; not attempted in this review.

## verdict: **ready_with_minor_fixes**

The slice is structurally sound and honestly bounded. The two major findings (lane independence documentation and prompt/output reproducibility metadata) are documentation gaps, not overclaims — the evidence states themselves are conservative. Fix the frontmatter on prompt/output files, name the three lanes explicitly with their dependency structure, and add the parity summary line. No blockers remain for a docs-only stress slice.
