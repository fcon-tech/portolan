# Spec 072 Audit: Findings Report

## Critical Findings

**None identified.** The spec suite demonstrates strong evidence-state discipline throughout.

---

## Major Findings

### M1. Artifact set is heavily skewed to third-party test fixtures, not Bigtop artifacts

**Files:** `plan.md` → Producer Results, `ledger` → Selected Scope

5 of 9 artifacts are well-known third-party JARs (log4j, gson, commons-logging, slf4j) sitting in `src/test/resources/`. These are *bundled dependencies*, not Bigtop-compiled code. Two more are tiny Hive UDF test fixtures (DummyUDF). Only `cachedir.jar` is marginally a Bigtop artifact.

**Impact:** The "3 of 15 repos have artifacts" framing in `plan.md` and `ledger` is technically accurate but misleading in implication. A reader could infer Bigtop projects' own compiled output was analyzed. The spec should state upfront that ~78% of dependency rows (190/289 from log4j alone) come from analyzing a bundled third-party logging library's internal bytecode dependencies — not from Bigtop's dependency structure.

**Recommendation:** Add a frank qualifier to `plan.md` Producer Results and `ledger` Results: *"The artifact set is predominantly bundled third-party test fixtures and UDF samples, not Bigtop-compiled production output. Row counts reflect third-party internal bytecode wiring, not Bigtop architecture."*

### M2. SC-002 is unachievable for the full artifact set as written

**File:** `spec.md` → SC-002

> `jdeps` exits `0` for **all** assessed existing artifacts.

`cachedir.jar` produces 0 dependency rows and 0 unresolved rows. While `jdeps` exits `0`, it effectively contributes no evidence. The success criterion doesn't distinguish "exited 0 with meaningful output" from "exited 0 vacuously."

**Recommendation:** Amend SC-002 to: *"jdeps exits 0 for all assessed artifacts; at least N of 9 emit nonzero dependency rows"* or add a separate SC for minimum evidence density.

### M3. "Top target modules/packages" table in plan.md is a jdeps artifact cross-contamination artifact

**File:** `plan.md` → Top target modules/packages

The table lists `63 log4j-1.2.17.jar` and `17 gson-2.2.jar` as "target modules/packages." These are the *originating JAR names* appearing in `jdeps -verbose:package` output as the source column, not meaningful dependency targets. Including them in a "top target modules" table conflates source artifacts with dependency targets.

**Impact:** A reader would interpret this as "Bigtop depends on log4j 63 times" when it actually means "jdeps emitted 63 rows where the package source is log4j's own internal packages depending on something else."

**Recommendation:** Relabel to "Top originating packages/modules in jdeps output" or drop the self-referencing JAR-name entries and show only the JDK module and `not found` rows.

## Minor Findings

### m1. Ledger "Allowed Wording" understates the evidence

**File:** `ledger` → Allowed Wording

The allowed wording block is well-constructed, but it could be stronger by noting the artifact *types* (third-party test fixtures + UDF samples) to pre-empt the overclaiming concern raised in M1. Currently the allowed wording is silent on artifact provenance.

### m2. plan.md Evidence Boundary says "beyond Syft/CycloneDX and beyond ctags-only" without qualification

**File:** `plan.md` → Evidence Boundary

> It is beyond Syft/CycloneDX and beyond ctags-only source tagging.

This is directionally true (jdeps provides internal package wiring that SBOM tools don't), but the comparison implies a qualitative leap. The actual delta is package-level bytecode dependencies on 9 small artifacts — a modest addition. A qualifying phrase like "in the narrow scope assessed" would be more precise.

### m3. tasks.md T010 references file paths not included in the spec directory

**File:** `tasks.md` → T010

The cursor stress output references `stress/cursor-existing-artifact-jdeps-prompt-2026-06-02.md` and `stress/cursor-existing-artifact-jdeps-output-2026-06-02.md`. The output file is included in the provided slice, but the prompt file is not. This is likely just a slice boundary issue but worth noting for completeness.

### m4. FR-008 references "Cursor stress" as a gate but the review document is self-assessed

**File:** `spec.md` → FR-008

> Cursor stress and independent review MUST preserve the evidence boundary

The Cursor stress output (`cursor-stress-ledger-2026-06-02.md`) is thorough and honest, but FR-008 frames it as a gate rather than a recorded self-assessment. The three independent non-GPT review lanes (T011) remain open, so this gate is only partially satisfied.

### m5. No path safety concern but no explicit path sanitization documented

**File:** `ledger` → Tool, `plan.md` → Producer Command

Artifact paths are relative and reasonable. The `jdeps` command uses a hardcoded Linuxbrew path and operates on local artifacts. No injection risk is apparent, but the spec doesn't document that artifact paths were validated (e.g., no symlinks escaping the selected roots, no path traversal). This is minor given the read-only scope.

**Overall assessment:** The spec suite is commendably disciplined on evidence boundary honesty — the allowed/disallowed wording blocks and explicit `cannot_verify` classifications are models of restraint. The main gap is that the suite is *so honest about the boundary* that it undersells how much of the "verified" evidence comes from analyzing third-party libraries' internal bytecode rather than Bigtop's own code. Fixing M1 and M3 would close the remaining honesty gap between what was proven and what the artifact provenance actually represents.
