# Pre-Implementation Review: Spec 053 — Language-Agnostic Producers

## Findings

### Critical

**C-1: Stacked-branch risk without merge gate**

PR #29 (spec 052) is open, non-draft, checks pass, but lacks review approval and merge approval. The merge-base of the 053 branch with the 052 branch is exactly the 052 head (`27ccbb9`), meaning 053 is a pure stack. However, the packet explicitly states "no implementation should start until this review is dispositioned and the branch policy is recorded" — yet no branch policy decision is recorded in the packet itself. The review packet is asking for a policy recommendation without having committed to one.

- **Impact:** If implementation begins and PR #29 requires changes, the 053 branch must be rebased, potentially invalidating implementation work and creating confusing review history.
- **Recommended fix:** Record an explicit branch policy before any implementation: *"No commits to codex/053 until PR #29 is merged and codex/053 is rebased onto origin/main through the merge commit."* This review should output that policy as part of its verdict.

### Major

**M-1: Spec scope boundary allows silent scope creep into scanner orchestration**

FR-002 calls for a "producer-family recommendation surface mapping blocked claims to needed local producer families." The recommendation record shape lists `candidate_tools` as an array of tool names. Combined with FR-007 (risky defaults are rejected/blocked/narrowed before default recommendation) and US2 (candidate evaluation records), there is a gradient from "listing options" to "evaluating and ranking options" that could drift into scanner orchestration or tool selection logic — which the product boundary explicitly excludes ("Portolan is a local, read-only navigation harness, not a coding harness").

The contract shapes as given are safe: every status is `not_assessed`, every decision is `not_assessed`. But the tasks list includes "producer candidate evaluation records" which implies an evaluation *process*, not just a schema for recording evaluations that were done externally.

- **Impact:** Without a hard guardrail, US2 implementation could introduce evaluation logic (checking if a tool is installed, probing its output format, scoring candidates) that crosses the product boundary into orchestration.
- **Recommended fix:** Add an explicit invariant to the spec: *"Evaluation records are authored by the operator or by a separate evaluation tool, never generated by Portolan itself. Portolan's role is to store, validate schema compliance, and surface evaluation records — not to perform the evaluation."* Alternatively, constrain US2 to fixture/schema only with a note that evaluation population is out of scope for this spec.

**M-2: FR-009/FR-010 contract guarantees lack enforcement mechanism**

FR-009/FR-010 state that no runtime topology or native language-semantics claims should be made from dependency, symbol, manifest, catalog, or recommendation records alone. This is a critical product-integrity constraint, but no task proposes a test, schema validation, or answer-contract guardrail that would *prevent* a future implementation from accidentally emitting such claims.

- **Impact:** The constraint is stated as a requirement but has no corresponding verification artifact. If recommendation records evolve to carry language-specific claim fields or runtime topology inferences, there is no automated check to catch the violation.
- **Recommended fix:** Add a task item under "Final" or "Foundational contract" that creates a schema constraint or answer-contract test asserting that producer-recommendation, producer-evaluation, and producer-coverage records do not carry fields that constitute runtime topology claims or language-semantics claims. A simple deny-list of field names or a schema `additionalProperties: false` with explicit allow-list would suffice.

**M-3: Coverage record scope ambiguity for mixed-language repositories**

FR-008 requires mixed-language coverage reported by repository and evidence family, including partial/off-scope coverage. The coverage record shape includes `scope: "repository"` and a `status` field, but there is no field for language multiplicity or per-language status within a repository. A monorepo with Python, TypeScript, and PHP would produce three coverage records for the same `repository_id` and `family`, with no way to distinguish which language is covered and which is partial.

- **Impact:** Implementers will either (a) collapse all languages into one record per repository+family (losing the granularity FR-008 requires), or (b) add an ad-hoc language field that wasn't spec'd, creating schema drift.
- **Recommended fix:** Either add an optional `languages_in_scope` field to the coverage record and clarify that one record per repository+family covers all languages (with partial coverage noted in `reason`), or specify that the coverage unit is repository+family+language and add `language` to the record key. The spec should choose one and document it.

### Minor

**m-1: Candidate tool list is already stale-dated by nature**

The recommendation record example lists `["scip", "lsif", "serena", "sourcebot", "zoekt"]` as candidate tools for symbol-index. These are reasonable as of the spec date, but the spec does not describe the maintenance process for this list or whether the list is intended to be exhaustive, illustrative, or operator-configurable.

- **Impact:** Low. If the list is illustrative (as the "options, not verified support" language suggests), this is fine. But a future reader may treat it as canonical.
- **Recommended fix:** Add a note to the spec or contract schema: *"candidate_tools is an illustrative list of options known at record creation time. It is not exhaustive and does not imply endorsement or verification."*

**m-2: No explicit `not_assessed` → assessed transition trigger**

FR-004 says recommendations preserve current weak states until local producer output is present and normalized. The contract uses `not_assessed` as the default. But the spec does not define what event transitions a recommendation from `not_assessed` to any assessed state, or who/what performs the transition.

- **Impact:** An implementer may create the transition logic (which again risks crossing into orchestration) or may leave the state permanent, making the field useless.
- **Recommended fix:** Add a single sentence: *"State transitions on recommendation and evaluation records are performed by the operator or by an external evaluation tool upon presentation of normalized producer output. Portolan does not autonomously transition these states."*

**m-3: Task list ordering does not reflect dependency on PR #29 artifacts**

The tasks are ordered setup → contract → US1 → US2 → US3 → final, but US1-US3 presumably depend on fixtures and schemas that may reference patterns or conventions established in spec 052 (dependency-symbol evidence import). If PR #29 changes those conventions post-merge, the 053 fixtures may need rework.

- **Impact:** Minor rework risk, mitigated by rebasing after merge.
- **Recommended fix:** Note in the task list that US1-US3 implementation should assume the 052 contract shapes as of the 053 branch point, with a post-rebase validation step after PR #29 merges.

---

## Verdict

**`pass_with_changes`**

The spec direction is sound, the product boundary is well-articulated, the contract shapes are honest (all-default `not_assessed`), and the anti-overclaim posture is correct. The findings above are addressable without re-architecting the spec.

## Recommendation

1. **Wait for PR #29 to merge before implementing.** Record the branch policy: no commits to `codex/053` until PR #29 is merged to `origin/main` and this branch is rebased. This eliminates stacked-branch risk entirely.
2. **Address M-1 through M-3** as spec amendments before implementation begins. These are specification gaps, not implementation details, and should be resolved at the spec level.
3. **After rebase and spec amendments,** implement in the order given (contract → US1 → US2 → US3 → final), with the additional guardrail tasks from M-2 inserted into the final phase.
