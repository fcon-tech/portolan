# Spec 073 — Independent Review: Bigtop Docker Provisioner Runtime Capture

## Findings

### F1. Provisioner exit code contradicts service reality
**Severity: Major**

The create command returned exit code 0, yet 4 of 6 roles failed or threw exceptions (NameNode FAILURE, ResourceManager 255/EXCEPTION, HistoryServer FAILURE, ProxyServer FAILURE, Datanode skipped). A zero exit code signals success to callers and CI, but the cluster was non-functional. This is a provisioner-level issue (Bigtop's `docker-hadoop.sh`), not a Portolan defect, but it materially affects evidence interpretation.

**Evidence:** Create exit 0; only NodeManager reached active/running. NameNode never formatted or started. ResourceManager exited 255.

**Recommendation:** Document explicitly in the evidence record that exit code 0 ≠ functional cluster. If Portolan ever gates downstream work on provisioner success, parse role-level status, not just the exit code.

---

### F2. Evidence scope is honestly bounded
**Severity: Minor (positive)**

The spec/backlog correctly scopes what was captured: Docker container, network, image metadata, role attempt outcomes, process evidence, and cleanup verification. Full topology, dependency graphs, symbol/reference graphs, call graphs, and enterprise parity are marked `cannot_verify`. This is conservative and appropriate.

**Evidence:** Spec wording acknowledges partial runtime-visible evidence; no overclaiming.

---

### F3. Cleanup verification is thorough
**Severity: Minor (positive)**

Post-destroy checks confirm: no residual containers, no residual networks, no volumes, no generated config, no `.provision_id`, no target repo residue. For a local-first Docker provisioner, this is the correct cleanup surface.

**Evidence:** Destroy exit 0; post-destroy residue checks all empty.

---

### F4. No Portolan code changes — boundary respected
**Severity: Minor (positive)**

The PR is docs/evidence only. No runtime code, no configuration, no pipeline logic was modified. The evidence capture is observational.

---

### F5. Approval audit trail present
**Severity: Minor (positive)**

User approved with `разрешаю` at a precise timestamp. The approved commands and their arguments are recorded verbatim.

---

### F6. Partial service failure limits evidentiary weight
**Severity: Major**

Only NodeManager was confirmed running. NameNode (HDFS) never started; ResourceManager crashed; Datanode was skipped. The captured runtime evidence covers a single YARN service in isolation, not a functioning Hadoop cluster. Any downstream spec or test that assumes a working Bigtop deployment cannot rely on this capture.

**Evidence:** 1/6 roles active; HDFS layer entirely absent.

**Recommendation:** If a future spec requires a functional cluster capture, re-provision with diagnostics (e.g., check HDFS formatting, DNS, memory limits) and capture role-level Puppet agent output.

## Verdict

**`cannot_verify`** (for full Bigtop cluster behavior) / **`verified`** (for the narrow claim: "a single-node Bigtop Docker provisioner was run, captured, and cleaned up with the stated partial results").

The evidence slice is sufficient to confirm the **mechanics** of provision → capture → destroy. It is **not** sufficient to confirm Bigtop deployed a functional Hadoop cluster. The spec correctly acknowledges this boundary.

## Not Assessed

- Symbol/reference graph, call graph, or code-level analysis.
- Enterprise/human parity of the provisioner.
- Root cause of NameNode/ResourceManager/HistoryServer/ProxyServer failures.
- Reproducibility on different hosts or with different Bigtop versions.
- Network-level traffic capture or inter-container communication evidence.
