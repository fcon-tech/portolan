# Product Contract: Captain Atlas

## Job Story

When I am responsible for a large software estate, I want to give an AI coding
agent a Portolan link and a path or repository set, so that the agent builds an
interactive atlas and helps me understand what exists, how it is connected, what
is risky, and where I can drill down.

## Product Promise

Portolan turns an agent from a code reader into a landscape navigator.

The product is not a report generator. It is a local atlas generator: an agent
can prepare the target, run bounded local discovery, generate a polished app,
and use the resulting bundle to answer questions.

## Non-Goals

- Build another coding harness.
- Replace Cursor, OpenCode, Codex, Sourcegraph, Backstage, or Understand
  Anything.
- Ask the agent to create a new UI from scratch on every run.
- Lead with evidence discipline as the product value proposition. Evidence
  labels are guardrails underneath the atlas, not the reason the captain opens
  Portolan.
- Start services, mutate target repositories, export source, read secrets, or
  call vendor APIs by default.
- Optimize for mobile interfaces.

## Shared Definitions

- **Captain**: the human owner of the software estate.
- **Target ecosystem**: one repo, many repos, a monorepo, or a local directory
  containing multiple projects.
- **Atlas app**: local web app generated by Portolan for a specific target.
- **Atlas bundle**: machine-readable data backing the app and agent Q&A.
- **Drill-down**: navigation from map node, relationship, risk, or answer to
  local files, facts, reports, commands, and known gaps.
- **First run**: the clean path from a Portolan link/path and target ecosystem
  to an opened atlas.

## Common BDD

```gherkin
Feature: Captain receives a useful atlas

Scenario: Captain starts from an agent prompt
  Given a target ecosystem with multiple repositories
  And a Portolan link or local checkout path
  When the captain asks an agent to build an atlas
  Then the agent can discover the Portolan instructions
  And the agent can run the documented first-run path
  And the captain receives a local atlas app URL or launch command

Scenario: Atlas explains the landscape
  Given a completed atlas bundle
  When the captain opens the atlas app
  Then the first screen identifies the target, repo count, major components, and coverage gaps
  And the map shows repos, components, and important relationships
  And the app ranks risks and questionable areas with drill-down

Scenario: Agent answers from the atlas
  Given a completed atlas bundle
  When the captain asks a follow-up question
  Then the agent queries the bundle before answering
  And the answer cites local atlas nodes, facts, or files
  And unsupported claims are marked as unknown, partial, or cannot verify

Scenario: Path repeats on a different ecosystem
  Given a second OSS ecosystem with a different stack
  When the agent runs the same first-run path
  Then Portolan produces an atlas without target-specific choreography
  And the scorecard records differences in coverage, runtime, failures, and usefulness
```

## Decision Gate

For every work package:

- Simpler/Faster: what is the shortest path to a useful atlas?
- Blocking Edge Cases: what prevents that answer on large multi-repo estates?
- Existing Open Source: should Portolan use, wrap, or defer to an existing tool?

## Implementation Slice

- Owned surfaces: installed harness instructions, target-local wrappers, atlas
  bundle contract, local viewer, bounded query/Q&A tools, handoff artifacts,
  acceptance gates, and claim-boundary docs.
- First vertical slice: an agent starts from `PORTOLAN` and `TARGET_ROOT`,
  installs target-local commands, builds a local atlas bundle, opens or hands
  off the viewer, runs bounded Q&A/selected-code checks, and writes a captain
  handoff.
- Artifacts: `$BUNDLE_DIR/captain-atlas-scorecard.json`,
  `$BUNDLE_DIR/captain-qna-eval.json`, `$BUNDLE_DIR/captain-handoff.json`, and
  `$BUNDLE_DIR/captain-handoff.md`.
- Verify: `scripts/portolan-product-acceptance.sh --northstar` with existing
  Bigtop and second-OSS relationship-bearing bundles; focused viewer and corpus
  smokes for UI drill-down; live Cursor/OpenCode captain runtime lanes when
  available.
- Out of scope: hosted collaboration, account systems, credential storage,
  daemon behavior, target service startup, broad enterprise completeness, and
  unapproved network/vendor/runtime capture.

## Global Acceptance Scorecard

| Dimension | Pass Condition |
| --- | --- |
| Agent autonomy | Agent completes first run without Portolan-author intervention. |
| Install reliability | Target-local commands and app launch work from documented instructions. |
| Atlas usefulness | Captain can identify repos, components, relationships, risks, and gaps from the app. |
| Drill-down | Map, findings, and answers route to local facts/files/reports. |
| Scale | Large corpora produce partial usable output instead of silent failure. |
| Portability | Same route works on Bigtop and at least one non-JVM OSS ecosystem. |
| OSS honesty | Capabilities better served by existing tools are wrapped or dropped. |

## Output For Each Agent

Each parallel agent must produce:

- a short verdict;
- commands or manual steps;
- screenshots or terminal evidence when relevant;
- changed files;
- BDD scenarios passed or failed;
- top blockers;
- kill / pack / build recommendation.
